# 運用マニュアル: スクレイピングコード (scripts/) の自動デプロイと同期

dso_system の運用では、「基盤システム（本リポジトリ）」と「各サイトのスクレイピングコード（`scripts/` ディレクトリ）」が別々の Git リポジトリとして管理されるマルチリポジトリ構成を想定しています。

このドキュメントでは、スクレイピングコードの変更を自動で本番環境の Worker に反映させる（CI/CD）運用方法について解説します。

## 1. 自動同期の仕組み

GitHub Actions や外部の CI ツールは一切使用しません。
**「Prefect Worker がスクレイピングを開始する直前に、自ら `git pull` を行う」** というアプローチによって自動同期を実現しています。

### 同期フロー

1. 開発者が `scripts/` リポジトリ（各サイトのクローラー実装など）の変更を GitHub に Push します。
2. Prefect Server のスケジュール設定に基づき、Worker コンテナがタスクを受注します。
3. `bin/run_flow.py` または `bin/retry_and_cleanup.py` が実行されると、**一番最初の Task** として `pull_scripts_repo_task` が自動的に実行されます。
4. この Task 内で `git -C scripts pull` コマンドが実行され、GitHub から最新のスクレイピングコードが Worker 内にダウンロード（上書き）されます。
5. その直後、常に最新化されたコードを用いてスクレイピング本体が実行されます。

## 2. 運用設定方法（初回のみ）

自動同期を有効にするためには、本番環境の Worker コンテナ内で `scripts/` ディレクトリが Git リポジトリとして認識されている必要があります。

### 前準備（GitHubの設定）
あらかじめスクリプトのリポジトリ（例: `dso_system-Scripts`）をGitHub上に作成し、ソースコードを `main` ブランチに Push しておきます。
また、本番環境の Worker がプログラムから自動で Pull できるように、**Personal Access Token (PAT)** を発行してください。
1. GitHubの `Settings` > `Developer settings` > `Personal access tokens (classic)` から新しいトークンを生成します。
2. スコープ（権限）は `repo` (フルコントロール、または読み取り権限) にチェックを入れます。
3. トークンをコピーしておきます。

### Windows環境で自動セットアップする場合（推奨）

本番運用環境が Windows 11 の場合、バッチファイルで自動的に初期設定が行えます。

1. プロジェクトルートの `.env` ファイルに以下の行を追加し、先ほど発行したトークンを埋め込みます。
   ```env
   # .env
   SCRIPTS_REPO_URL=https://<あなたのGitHubトークン>@github.com/STREAM-inc/dso_system-Scripts.git
   ```
2. Docker Desktop等でコンテナ（`worker`等）が起動している状態にします。
3. プロジェクトルートにある `setup_scripts_git.bat` をダブルクリックして実行します。
4. 全自動でコンテナ内部に入り、`git init` や `git pull` などの設定が完了します。

### 手動でセットアップする場合（Linux / macOS 等）

本番サーバー（または Docker コンテナ）上で以下の設定を行ってください。

```bash
# Worker コンテナに入る
docker compose exec worker bash

# scripts/ ディレクトリに移動
cd /app/scripts

# Git リポジトリとして初期化し、リモートリポジトリを登録する
git init
git remote add origin https://github.com/STREAM-inc/dso_system-Scripts.git (※実際のリポジトリURL)

# メインブランチを Pull して紐付ける
git pull origin main
git branch -M main
git branch --set-upstream-to=origin/main main
```

※ **注意**: GitHub へのアクセス権限（SSHキーの配置や、HTTPSでのPersonal Access Tokenの設定）が本番環境の Worker に対して正しく設定されている必要があります。

## 3. メリット

* **無料**: GitHub Actions などのクラウドリソース（ビルド時間）を全く消費しません。
* **シンプル**: Dockerビルドの再実行や、コンテナの再起動は一切不要です。
* **確実**: `git pull` コマンドがエラーになった場合は、Prefect 上のダッシュボードでその Flow 全体が「Failed」として即座に検知できます。
* **メンテナンスにも適用**: S3再送やクリーンアップ処理の直前にも最新の基盤ロジック群を確実に読み込む方針としています（設計上 `scripts/` に限定されないユーティリティ追加等もカバーできます）。

## 4. トラブルシューティング

Prefect Dashboard でタスクが失敗し、ログに `scripts リポジトリの同期に失敗しました` と表示されている場合、以下の原因が考えられます。

* **アクセス権限エラー**:
  Worker 上の Git 認証情報（SSH キーや Token）の有効期限が切れているか、未設定です。
* **コンフリクト（競合）エラー**:
  本番の `scripts/` ディレクトリ内で一時的に手動でコードを書き換えてしまった場合、`git pull` がマージエラーを起こして止まります。
  直すには Worker コンテナに入り、`git reset --hard HEAD` 等で変更をリセットしてください。
* **Git 初期化漏れ**:
  `scripts/` フォルダの中に `.git/` が存在しない場合、システムは「Git管理されていない」と判断し Pull をスキップ（正常終了扱い）して古いコードのまま実行を続けます。
