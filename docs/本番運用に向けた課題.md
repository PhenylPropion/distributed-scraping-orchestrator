# dso_system 本番運用に向けた残課題と対策案 (Production Readiness)

現状の dso_system は非常に優れたアーキテクチャ（Prefect 3.x によるオーケストレーション、Playwright による動的スクレイピング、S3 へのデータ連携、メタデータの SSOT 化）を持っています。
ここから「開発環境で動く確固たるシステム」を「本番環境で安定して自律稼働するシステム」に引き上げるために解決すべき課題を洗い出しました。

---

## 1. インフラ・デプロイメントの課題 (Infrastructure & Deployment)

### 1.1 GitHub → Worker の自動同期 (CI/CD)
**【現状】** Workerコンテナは `./src` や `./scripts` をボリュームマウントして動作しています。本番用の専用サーバー（EC2等）で動かす場合、ソースコードの更新をどう反映するかが未定義です。
**【対策案】**
- **案A (Pull-based):** Worker コンテナの起動時、または定期的に `git pull` を実行する仕組みを入れる。
- **案B (Push-based / Image build):** GitHub Actions を導入し、main ブランチ更新時に Docker イメージを自動ビルドしてコンテナレジストリ（GHCR や ECR）に Push し、本番サーバーで `docker compose pull && up -d` を行う。**（推奨: 再現性が高くロールバックが容易）**

### 1.2 コンテナのヘルスチェックと自動復旧
**【現状】** [docker-compose.yml](file:///Users/eigou/Desktop/dso_system/docker-compose.yml) に `database` と `server` のヘルスチェックはありますが、`worker` のヘルスチェックがありません。Playwright がメモリリークを起こしてゾンビ化した場合、Prefect Server からは「Offline」になっても再起動されません。
**【対策案】**
- Worker コンテナにヘルスチェックを追加し、応答がなければ Docker デーモンが自動的にコンテナを再起動（`autoheal` または Docker ネイティブの再起動）するようにする。

## 2. 運用・監視の課題 (Operations & Monitoring)

### 2.1 スケジュールの本番登録体制
**【現状】** [bin/deploy.py](file:///Users/eigou/Desktop/dso_system/bin/deploy.py) で [scripts/sites.yml](file:///Users/eigou/Desktop/dso_system/scripts/sites.yml) からスケジュールを登録できますが、「本番環境の cron 設定」のレビュープロセスがありません。
**【対策案】**
- Target サイトごとの負荷分散（全サイトが深夜0時に同時起動してワーカーがパンクしないか）を考慮し、[scripts/sites.yml](file:///Users/eigou/Desktop/dso_system/scripts/sites.yml) の `schedule` を分散させる。
- 月次・週次などの長期間隔スケジュールのテスト運用（手動トリガーでの挙動とリソース消費の確認）。

### 2.2 S3 転送失敗時のリカバリと再送制御
**【現状】** [run_flow.py](file:///Users/eigou/Desktop/dso_system/bin/run_flow.py) 内で [upload_to_s3_task(retries=2, retry_delay_seconds=10)](file:///Users/eigou/Desktop/dso_system/bin/run_flow.py#135-150) が設定されており、一時的なネットワークエラーには耐えられますが、クレデンシャル有効期限切れ等で完全に失敗した場合、ローカルの [output/](file:///Users/eigou/Desktop/dso_system/tests/test_pipeline.py#26-30) フォルダに CSV が残ります。
**【対策案】**
- 過去の失敗分（`output/*.csv`）を検知して一括再送するリカバリ用 Flow（またはスクリプト）を作成する。
- ローカルストレージ圧迫を防ぐための定期的なクリーンアップ（例: 転送成功から7日経過したローカルCSVの削除）。

## 3. スクレイピング品質とデータ整合性の課題 (Data Quality & Idempotency)

### 3.1 IP ブロック対策とプロキシ (IP Rotation & Anti-Bot)
**【現状】** 現在アクセス元の IP は固定（Worker サーバーの IP）です。本番稼働でアクセス頻度が増えると、大量アクセスにより相手サーバーから 403 Forbidden などの IP ブロックを受けるリスクが高まります。
**【対策案】**
- 動的サイト（Playwright）および静的サイト（Requests）の双方で、商用プロキシプロバイダ（ScraperAPI, BrightData, Smartproxy等）を経由させる設定を導入する。
- 失敗時の 403 やキャプチャ要求（Cloudflare ターンストイル等）を検知した場合のリトライロジック強化。

### 3.2 差分抽出とデータ重複（Idempotency）
**【現状】** 現在のクローラーは毎回全件を取得（Full Scraping）しています。日次で取得する場合、全く同じデータが何度も S3 に転送されます。
**【対策案】**
- 相手サイトへの負荷を考え全件取得は継続しつつ、生成されたCSVファイル名にタイムスタンプを付与（実装済み）して S3 側で差分を吸収するアーキテクチャで運用を合意する。
- 万が一途中でエラー終了した際の「重複・欠損」をどう扱うか仕様を明確にする（現状は中途半端なCSVも生成されてしまうため、Flow 終了時に完了確認ログを入れるなどの運用が必要）。

## 4. セキュリティとリソース管理の課題 (Security & Resource Management)

### 4.1 シークレット管理機能の本番適用
**【現状】** AWS S3 の認証情報等は [.env](file:///Users/eigou/Desktop/dso_system/.env) ファイルで管理されています。
**【対策案】**
- 本番サーバーのデプロイメント時に [.env](file:///Users/eigou/Desktop/dso_system/.env) を手動配置する運用で良いか確認する。
- Prefect 3.x の機能である **Prefect Blocks (Secret Block)** を使用すれば、AWS クレデンシャルを Prefect Server 側（PostgreSQL内、暗号化済）で一元管理でき、Docker の [.env](file:///Users/eigou/Desktop/dso_system/.env) を持たせる必要がなくなるため本番運用としては推奨されます。

### 4.2 メモリリーク対策と OOM (Out Of Memory) キル
**【現状】** クローラーの [BaseCrawler](file:///Users/eigou/Desktop/dso_system/src/framework/base.py#33-294) で一時ストリーミングを採用しているためメモリ効率は良いですが、Playwright（[DynamicCrawler](file:///Users/eigou/Desktop/dso_system/src/framework/dynamic.py#18-77)）は数千ページの遷移を行うとブラウザプロセスのメモリが肥大化する傾向があります。
**【対策案】**
- [docker-compose.yml](file:///Users/eigou/Desktop/dso_system/docker-compose.yml) で worker にハードメモリ制限（例: `deploy: resources: limits: memory: 4G`）を設ける。
- 動的クローラーで一定ページ数（例: 500ページ）を処理するごとに `context` を再生成する仕組みを将来的に検討する。

## まとめ：優先アクションプラン (Next Actions)

直近で実装・検討すべき「P0~P1」のアクションは以下の 3 点です。

1. **シークレット管理の移行:** [.env](file:///Users/eigou/Desktop/dso_system/.env) の AWS キー等を Prefect Blocks に移行して運用を堅牢にするか検討。
2. **GitHub Actions の構築:** `docker build` と本番サーバーへのデプロイを自動化する CI/CD パイプラインの作成（本番環境のサーバー実体が確定している場合）。
3. **ローカルCSVのクリーンアップスクリプト:** S3アップロード完了後に、一定期間経過したローカル一時ファイルを削除する仕組みの追加。
