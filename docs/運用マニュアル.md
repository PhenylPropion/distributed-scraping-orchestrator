# dso_system 運用マニュアル

> **対象:** スクレイパーを実際に動かす運用者
> **前提:** Docker が起動済みであること

---

## 1. 実行方法の全体像

dso_system には目的別に 3 つの実行方法がある。

| 方法 | コマンド | 用途 | Worker使用 |
| ---- | ------- | ---- | ---------- |
| **A. ローカル直接実行** | `python bin/run_flow.py` | 動作確認・開発テスト | ✗（このPCで実行） |
| **B. 特定サイト投入** | `python bin/trigger.py` | 手動で特定サイトをWorkerに投入 | ✓ |
| **C. 自動定期実行** | Prefectが自動実行 | `schedule` 設定済みサイトの定時実行 | ✓ |

---

## 2. 事前準備

### Docker 起動

```bash
# Server（Prefect管理サーバー + DB）+ Worker を全起動
docker compose --profile server --profile worker up -d

# 起動確認
docker compose ps
```

正常な状態:

```text
dso_system-database-1   ... Up (healthy)
dso_system-server-1     ... Up (healthy)
dso_system-worker-1     ... Up
```

### Prefect UI

起動後は [http://localhost:4200](http://localhost:4200) で管理画面にアクセスできる。
本番サーバーの場合は `http://サーバーIP:4200`。

---

## 3. 方法A: ローカル直接実行（`run_flow.py`）

Worker を使わず、**このPC上で直接** スクレイパーを実行する。
開発テストや1件だけ今すぐ動かしたいときに使う。

```bash
# sites.yml に登録されているサイトを site_id で指定
python bin/run_flow.py --site-id eaidem
python bin/run_flow.py --site-id refonavi
```

**実行結果:**

- CSV が `output/` フォルダに生成される
- Prefect UI に実行ログが記録される

> **注意:** このPCが重くなるのでWorkerに任せたい場合は 方法B を使う。

---

## 4. 方法B: Workerへの手動投入（`trigger.py`）

**Prefect の Work Pool を通じて Worker に実行を依頼する。**
Worker が仕事を受け取り、CSVを生成する。Worker が複数台あれば並列実行される。

### 事前条件（初回 or sites.yml 変更時のみ）

Worker にデプロイメントを登録する：

```bash
docker compose exec worker python bin/deploy.py

# 内容確認だけしたい場合（登録・削除・投入は行わない）
docker compose exec worker python bin/deploy.py --dry-run
```

`deploy.py` は以下を自動で行う：

| 処理 | 内容 |
| ---- | ---- |
| **新規登録** | `enabled: true` の全サイトのデプロイメントを Prefect に登録・更新 |
| **初回自動投入** | **初めて登録されたサイト**のフローランをワークプールに即時投入 |
| **古いデプロイ削除** | `sites.yml` から削除・無効化されたサイトのデプロイメントを Prefect から自動削除 |

### 実行コマンド

```bash
# 1サイトだけ投入
docker compose exec worker python bin/trigger.py eaidem

# 複数サイトを一括投入（スペース区切り）
docker compose exec worker python bin/trigger.py eaidem shinq_compass refonavi

# sites.yml の全 enabled サイトを一括投入
docker compose exec worker python bin/trigger.py --all

# 環境を明示指定（デフォルトは dev）
docker compose exec worker python bin/trigger.py --all --env prod

# 完了まで待機してから終了する
docker compose exec worker python bin/trigger.py eaidem --wait
```

**出力例:**

```text
Prefect API : http://server:4200/api
環境        : dev  |  プール: dev-pool

✅ 投入完了: eaidem-dev  (イーアイデム)
   → http://server:4200/flow-runs/flow-run/xxxx-xxxx

投入: 1件  失敗/スキップ: 0件
Prefect UI で進捗を確認: http://server:4200/flow-runs
```

### 本番サーバーに向ける場合（開発PCから投入）

```bash
# 本番サーバーの Prefect API に向ける
export PREFECT_API_URL=http://本番サーバーIP:4200/api
export dso_system_ENV=prod

docker compose exec worker python bin/trigger.py --all --env prod
```

## 5. 方法C: 自動定期実行（スケジュール）

`sites.yml` で `schedule` を設定したサイトは、Prefect が自動実行する。

```yaml
# scripts/sites.yml
- site_id: eaidem
  name: "イーアイデム"
  module: "jobs.eaidem"
  url: "https://..."
  schedule: "0 3 1 * *"   # 毎月1日 午前3時に自動実行
  enabled: true
```

**cron 式の書き方:**

```text
┌───── 分 (0-59)
│ ┌───── 時 (0-23)
│ │ ┌───── 日 (1-31)
│ │ │ ┌───── 月 (1-12)
│ │ │ │ ┌───── 曜日 (0-7, 0と7は日曜)
│ │ │ │ │
0 3 1 * *   → 毎月1日 午前3時
0 2 * * 0   → 毎週日曜 午前2時
0 1 * * *   → 毎日 午前1時
```

スケジュールを変更した場合は `deploy.py` を再実行して反映する：

```bash
docker compose exec worker python bin/deploy.py
```

> **`schedule: null`** を設定したサイトはスケジュール実行されない（`trigger.py` または `run_flow.py` で手動実行のみ）。

## 6. 新しいサイトを追加する手順

### Step 1: スクレイパーコードを作成

```text
scripts/sites/{カテゴリ}/{site_id}.py
```

開発方法の詳細は [`開発マニュアル.md`](開発マニュアル.md) を参照。

### Step 2: sites.yml に登録

```yaml
# scripts/sites.yml の末尾に追記
- site_id: mysite            # 英語ID（ユニーク必須）
  name: "マイサイト"           # CSV名・ログに表示される日本語名
  module: "portal.mysite"   # scripts/sites/ からのパス（ドット区切り）
  url: "https://example.com"
  schedule: null             # null=手動のみ、cron式=自動定期
  enabled: true
```

### Step 3: デプロイメント登録（Workerに反映）

```bash
docker compose exec worker python bin/deploy.py
```

**初めて登録するサイトは、このコマンドだけでワークプールへの投入まで自動で行われる。**
実行ログに以下のように表示されれば投入完了：

```text
🚀 初回登録サイト (1 件) をワークプールに投入中...
   🚀 投入: マイサイト (mysite)-dev
✅ 1 件のフローランを作成しました
```

### Step 4: 動作確認

```bash
# ローカルで先に動作確認してから deploy.py を実行したい場合
python bin/run_flow.py --site-id mysite
```

> **注意:** `run_flow.py` で動作確認してから `deploy.py` を実行した場合、Step 3 の自動投入でもう一度実行される。
> 動作確認済みなら `sites.yml` に追記後すぐに `deploy.py` を実行してもよい。

## 7. 実行結果の確認

### CSV ファイル

```text
output/
├── 20260327_eaidem_イーアイデム_156件.csv   ← 取得データ
├── 20260327_refonavi_リフォーム評価ナビ_412件.csv
└── data_catalog.csv                         ← 全サイトのカタログ（自動生成）
```

ファイル名の形式: `YYYYMMDD_site_id_サイト名_N件.csv`

### Prefect UI で実行状況を確認

[http://localhost:4200/flow-runs](http://localhost:4200/flow-runs) で実行状況を確認できる。

| 表示 | 意味 |
| ---- | ---- |
| `Running` | 実行中 |
| `Completed` | 正常完了 |
| `Completed ⚠️` | 完了したが一部スキップあり（S3未設定など） |
| `Failed` | エラーで失敗（Teams通知が届く） |
| `Scheduled` | 定期実行の待機中 |

## 8. 2つの Flow の違い

dso_system には役割の異なる **2 つの Prefect Flow** がある。

| | `scrape_site_flow` | `maintenance_flow` |
| --- | --- | --- |
| **役割** | **本体** — 各サイトをスクレイピングし、CSV出力・S3転送・カタログ同期を行う | **裏方の掃除係** — S3未送信ファイルの再送と古いCSVの定期削除 |
| **定義ファイル** | `bin/run_flow.py` | `bin/retry_and_cleanup.py` |
| **実行タイミング** | サイトごとの cron スケジュール or 手動投入 | 毎日朝5時（固定スケジュール） |
| **パラメータ** | `site_id`, `site_name_ja`, `module_path`, `url` | `retention_days`（保持日数、デフォルト14日） |
| **タスク構成** | Git Pull → スクレイパー実行 → S3転送 → カタログ同期 | Git Pull → S3再送 → 古ファイル削除 |
| **Teams通知** | サイトごとに成功/失敗/警告を通知 | 再送・削除があった場合のみ通知 |

### いつ何が動くか

```text
毎月1日 3:00 AM  →  scrape_site_flow (eaidem)     ← スクレイピング本体
毎日    5:00 AM  →  maintenance_flow               ← S3再送 + 古ファイル削除
手動投入 随時     →  scrape_site_flow (任意サイト)   ← trigger.py or run_flow.py
```

> **補足:** `maintenance_flow` は S3 転送が一時的に失敗した場合の救済用。
> `scrape_site_flow` が正常に S3 転送を完了していれば、再送対象は0件となり何もしない。

## 9. よくある操作

### サイトを一時的に無効にする

```yaml
# sites.yml で enabled: false に変更
- site_id: mysite
  enabled: false   ← これで deploy.py 登録・trigger.py --all から除外される
```

変更後は `deploy.py` を再実行。**Prefect ダッシュボードのデプロイメントも自動で削除される。**

### サイトを完全に削除する

`sites.yml` からエントリを丸ごと削除して `deploy.py` を再実行。
Prefect 上のデプロイメントも自動で削除される。

### 特定サイトのスケジュールを外す

```yaml
schedule: null   ← cron式を消してnullにする
```

変更後は `deploy.py` を再実行。

### デプロイメント一覧を確認する

```bash
docker compose exec worker prefect deployment ls
```

### Worker のログを確認する

```bash
docker compose logs worker --tail=50
docker compose logs worker -f   # リアルタイム追跡
```

### 実行失敗したサイトを手動でリトライする

```bash
# run_flow.py で再実行
python bin/run_flow.py --site-id eaidem

# または trigger.py でWorkerに投入
docker compose exec worker python bin/trigger.py eaidem
```

## 10. 環境変数一覧

`.env` ファイルで設定する主な変数：

| 変数名 | 説明 | 例 |
| ------ | ---- | --- |
| `ORCHESTRATOR_IP` | Prefect サーバーの IP アドレス | `192.168.1.10` |
| `dso_system_ENV` | 環境区分（`dev` / `prod`） | `prod` |
| `WORKER_NAME` | Worker の表示名 | `worker-main` |
| `WORKER_LIMIT` | 1台のWorkerが同時に受け持つフローラン数（デフォルト: `1`） | `1` |
| `TEAMS_WEBHOOK_URL` | Teams 通知 Webhook URL | `https://...` |
| `S3_BUCKET_NAME` | CSV 転送先 S3 バケット名 | `dso_system-data` |
| `AWS_ACCESS_KEY_ID` | AWS 認証情報 | — |
| `AWS_SECRET_ACCESS_KEY` | AWS 認証情報 | — |

## 11. ファイル・ディレクトリ構成

```text
dso_system/
├── bin/
│   ├── run_flow.py       # A. ローカル直接実行
│   ├── trigger.py        # B. Workerへの手動投入
│   ├── deploy.py         # デプロイメント登録（初回・変更時）
│   └── retry_and_cleanup.py  # 定期メンテナンス（S3再送・古ファイル削除）
│
├── scripts/
│   ├── sites.yml         # サイト設定ファイル（ここを編集）
│   └── sites/            # スクレイパーコード（ここに追加）
│       ├── test/
│       ├── jobs/
│       └── portal/
│
├── output/               # 生成されたCSVファイル
├── logs/                 # 実行ログ
└── docs/                 # このマニュアルがあるフォルダ
```
