# 🌾 dso_system 開発者向け FAQ (よくある質問と回答)

dso_system基盤でクローラーを開発する際によくある質問、アーキテクチャの設計思想、およびベストプラクティスをまとめています。

---

## 設計・ファイル管理について

### Q. 「一覧のURL取得(fetch.py)」と「詳細データの抽出(scrape.py)」のように、複数ファイルに分けて開発・管理しても良いですか？

**A. 原則として「1サイト ＝ 1ファイル（1スクリプト）」で実装してください。**

ファイルを物理的に分割してしまうと、「取得したURLリストを一時ファイルに保存して、次のスクリプトに渡す」といったワークフローの依存関係をPrefect側でサイトごとに組む必要があります。また、実行順序の依存関係が複雑化し、エラー時のリトライ制御が破綻するバグの温床となります。このようなサイトが増えるほど、管理自体がスパゲティと化してしまいます。

**ベストプラクティス：メソッド分割**
サイトマップ取得 → 一覧ページネーション → 詳細抽出 といった複雑なフェーズがある場合は、ファイルを分けるのではなく、**1つのクラス内で内部メソッド（例: `_get_urls()`）としてフェーズを分割**して記述してください。
最後に `parse` メソッド内で詳細URLのループを回し、1件ずつ `yield` することで、コードの可読性を保ちながらインフラの恩恵（自動保存・エラー保護）を最大限に受けることができます。

## データの保存と出力について

### Q. 最後にまとめてCSV出力する設計だと取得状況が分かりづらいです。その都度出力にした方が良いのでは？

**A. ご安心ください。システムはすでに「都度出力（ストリーミング書き込み）」を行っています。**

皆様が `parse` メソッド内でデータを `yield` するたびに、裏側の `ItemPipeline` が即座に一時CSVファイルへストリーム書き込みを行っています。そのため、数十万件のデータを処理してもメモリを圧迫せず、万が一途中でクラッシュしても、そこまでに `yield` されたデータは保護されます。
現在の取得件数や完了予定時刻（ETA）などの取得状況は、ターミナルやPrefectダッシュボードのログでリアルタイムに確認いただけます。

## パフォーマンスと並列処理について

### Q. 処理速度を上げるため、システム側でデフォルトの並列処理（max_workers=5など）を入れてもらえませんか？

**A. システム全体での一律な並列化は、安全上の理由から意図的に見送っています。**

理由は大きく2つあります。

1. **対象サイトの保護とアクセス遮断の回避:** 多くのサイトはアクセス制限が厳しく、むやみに並列数を増やすと即座にIPバン（アクセス遮断）を受けるリスクがあります。
2. **コンテナリソースの保護:** 本システムはDocker環境上で複数サイトのクローラーを同時稼働させます。システム側で強制的にマルチスレッド化すると、コンテナのCPU・メモリを過剰に消費し、システム全体がダウンする危険性があります。

**並列化したい場合**
アクセス制限が緩く、データ量が膨大なサイトに限り、皆様が作成するクローラーの `parse` メソッド内部で `concurrent.futures` 等を用いて**個別に並列処理（マルチスレッド）を実装することは可能**です。対象サイトの規約やサーバー負荷に十分配慮した上でご活用ください。

## 運用・スケジュールについて

### Q. このツールにおいて「月次取得」などの定期実行はどのように設定・管理すればよいですか？

**A. クローラーのコード内にスケジュール処理を書く必要はありません。インフラ側で設定します。**

dso_systemでは、「いつ動かすか」というスケジュール管理はすべて設定ファイル（`scripts/sites.yml`）で一元管理しています。
開発者の皆様は通常通りクローラーを作成し、プロジェクト直下の `scripts/sites.yml` にサイト情報を登録する際、`schedule` 項目にCron式（例: 毎月1日の朝3時なら `"0 3 1 * *"`）を記述するだけで完了します。あとはシステム（Prefect）が自動的に月次実行をハンドリングします。

## エラーハンドリングについて

### Q. 通信エラー（404、タイムアウトなど）が発生したとき、クローラー全体が止まってしまいますか？

**A. デフォルトでは止まりません。** dso_system には「エラー耐性モード」(`CONTINUE_ON_ERROR`) が組み込まれており、デフォルトで有効 (`True`) です。

**自動スキップ対象のエラー:**

| エラー種別 | 例 | 挙動 |
|-----------|---|------|
| HTTP クライアントエラー | 404 Not Found, 403 Forbidden | `get_soup()` が None を返し、次の処理へ |
| HTTP サーバーエラー | 500, 502, 503, 504 | 自動リトライ 3 回 → それでも失敗なら None を返す |
| 接続エラー | タイムアウト、DNS解決失敗、接続拒否 | `get_soup()` が None を返し、次の処理へ |
| Playwright ナビゲーションエラー | ページ読み込み失敗、タイムアウト | `get_soup()` が None を返し、次の処理へ |
| アイテム処理中のエラー | HTML要素が見つからない等 | そのアイテムをスキップして次へ |

**即座に停止するエラー（CONTINUE_ON_ERROR の設定に関係なく常に停止）:**

| エラー種別 | 例 | 理由 |
|-----------|---|------|
| Schema 違反 (`ValueError`) | 未宣言のカラム名を使用 | 開発者のコードにバグがあるため |
| `KeyboardInterrupt` | Ctrl+C による手動停止 | ユーザーの意図的な停止 |

スキップされたエラーの件数は、完了時のログに `"完了: 120 件処理しました (⚠ 3 件のエラーをスキップ)"` のように表示され、Prefect ダッシュボードや Teams 通知にも警告として含まれます。

**エラー耐性モードを無効にしたい場合:**

厳密にすべてのデータが必要で、エラー時に即停止させたい場合は以下のように設定します:

```python
class MyScraper(StaticCrawler):
    CONTINUE_ON_ERROR = False  # エラー発生時に即座に停止
```
