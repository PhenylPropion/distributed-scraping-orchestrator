## 1. なぜ統括層が必要なのか？

前回の `詳細解説.md` で説明した通り、コア基盤（BaseCrawler, ItemPipeline 等）は
**「スクレイピングして CSV に保存する」** ところまでを担当します。

しかし、実際の運用ではこんな疑問が出てきます:

```
・いつ、どのスクレイパーが実行されたのか？
・実行は成功したのか、失敗したのか？
・失敗したらどこでエラーが出たのか？
・毎月1日に自動で実行したい
・ダッシュボードで一覧管理したい
```

これらは**スクレイピングそのものの仕事ではありません**。
「誰が」「いつ」「どのスクレイパーを」「どの URL に対して」実行するかを管理する仕事です。

この管理の仕事を担うのが **統括層（Orchestration Layer）** です。

```
┌──────────────────────────────────────────────────────────┐
│ 統括層 (run_flow.py / Prefect)                      　    │
│  「今日は sample を実行しろ。URL はこれだ」　　　　　　　　　　　　│
│  「実行結果をダッシュボードに記録しろ」                 　　　　 │
│  「失敗したらログに残せ」                            　　 　  │
└────────────────────────┬─────────────────────────────────┘
                         │ 呼び出し
                         ▼
┌──────────────────────────────────────────────────────────┐
│ コア基盤 (BaseCrawler / Pipeline / normalizer)            │
│  「スクレイピングして CSV に保存する」                        │
│  ※ 自分がどう管理されているかは知らない                        │
└──────────────────────────────────────────────────────────┘
```

> **ポイント:** コア基盤は Prefect の存在を**一切知りません**。
> `base.py` や `pipeline.py` のどこにも `prefect` という文字は出てきません。
> 統括層が一方的にコア基盤を「使っている」だけです。これが疎結合です。

## 2. Prefect とは何か

### 2.1 ひとことで言うと

**Python の関数に「管理機能」を後付けするフレームワーク** です。

普通の Python 関数 → Prefect のデコレータをつけるだけで:

- 実行ログが自動記録される
- 成功/失敗がダッシュボードに表示される
- リトライ（自動再試行）が設定できる
- スケジュール実行（毎日朝 9 時に自動実行など）ができる

### 2.2 基本概念: Flow と Task

Prefect には 2 つの核心的な概念があります:

```
Flow（フロー）= 仕事全体の流れ
  └── Task（タスク）= 仕事の中の1つのステップ
```

料理に例えると:

```
Flow: 「カレーを作る」（仕事全体）
  ├── Task 1: 「野菜を切る」
  ├── Task 2: 「肉を炒める」
  ├── Task 3: 「煮込む」
  └── Task 4: 「盛り付ける」
```

このシステムでは:

```
Flow: 「サイトをスクレイピングする」
  └── Task: 「指定されたクローラーを実行する」
```

現段階では Task は 1 つだけですが、将来こう拡張できます:

```
Flow: 「サイトをスクレイピングする」
  ├── Task 1: 「クローラーを実行する」      ← 今ここ
  ├── Task 2: 「CSV を NAS に転送する」     ← 将来追加
  └── Task 3: 「完了メールを送信する」      ← 将来追加
```

### 2.3 デコレータ: `@flow` と `@task`

デコレータとは、関数の前に `@` をつけて「機能を追加する」Python の仕組みです。

```python
# 普通の関数
def greet(name):
    print(f"こんにちは、{name}さん")

# Prefect の Task として登録された関数
@task
def greet(name):
    print(f"こんにちは、{name}さん")
```

**中身のコードは全く同じ** です。`@task` をつけるだけで、Prefect が
「この関数がいつ呼ばれたか」「成功したか」「何秒かかったか」を自動記録してくれます。

もう少し詳しく言うと、デコレータは **「関数を受け取って、新しい関数を返す関数」** です:

```python
# @task は、裏でこういうことをしている（イメージ）:

def task(func):
    def wrapper(*args, **kwargs):
        print(f"[Prefect] {func.__name__} 開始")     # ← 追加された機能
        result = func(*args, **kwargs)                 # ← 元の関数を実行
        print(f"[Prefect] {func.__name__} 完了")     # ← 追加された機能
        return result
    return wrapper

@task                            # greet = task(greet) と同じ
def greet(name):
    print(f"こんにちは、{name}さん")
```

### 2.4 `get_run_logger()` — Prefect 専用のロガー

```python
from prefect.logging import get_run_logger

logger = get_run_logger()
logger.info("なにかメッセージ")
```

これは Python 標準の `logging` と似ていますが、出力先が違います:

- `logging` → ターミナル（コンソール）に出力
- `get_run_logger()` → ターミナル **＋ Prefect ダッシュボード** に出力

ダッシュボードで実行ログを確認できるようになります。

## 3. 前提知識: 使っている Python の仕組み

### 3.1 `importlib` — モジュールの動的インポート

通常の `import` は **コードを書く時点で** 何をインポートするか決めています:

```python
from scripts.sites.sample import SampleScraper    # ← 「sample」が固定
```

しかし、このシステムでは **実行時にコマンドライン引数で** サイト名が決まります:

```bash
python run_flow.py --site sample       # ← 実行時に「sample」が決まる
python run_flow.py --site tablog       # ← 別の日は「tablog」かもしれない
```

`importlib` を使うと、**文字列から** モジュールをインポートできます:

```python
import importlib

# 通常の import（静的）
import scripts.sites.sample

# importlib を使った import（動的）— 文字列で指定できる！
site_name = "sample"                                   # ← 変数（実行時に決まる）
module = importlib.import_module(f"scripts.sites.{site_name}")
# => scripts/sites/sample.py が読み込まれる
```

### 3.2 `inspect` — クラスの中身を調べる

`inspect` モジュールはPython オブジェクトの内部情報を調べるツールです。

```python
import inspect

# モジュール内の全クラスを列挙する
for name, obj in inspect.getmembers(module, inspect.isclass):
    print(f"クラス名: {name}, 定義場所: {obj.__module__}")
```

`inspect.getmembers(module, inspect.isclass)` は:

- 第1引数: 調べたいモジュール
- 第2引数: フィルター条件（`inspect.isclass` = 「クラスだけ」）
- 戻り値: `(名前, オブジェクト)` のリスト

`sample.py` を調べると、こんなクラスが見つかります:

```python
[
    ("BaseCrawler",    <class BaseCrawler>),      # ← import で入ってきたクラス
    ("SampleScraper",  <class SampleScraper>),    # ← このモジュールで定義されたクラス
    ("Schema",         <class Schema>),            # ← import で入ってきたクラス
    ("StaticCrawler",  <class StaticCrawler>),    # ← import で入ってきたクラス
]
```

この中から「欲しいクラス」を見つけるために 2 つの条件でフィルターします:

```python
if issubclass(obj, BaseCrawler) and obj is not BaseCrawler:
    # ① BaseCrawler を継承している
    # ② BaseCrawler 自身ではない

    if obj.__module__ == module.__name__:
        # ③ このモジュールで定義されたクラスである
        #    （import で入ってきた StaticCrawler 等を除外）
```

| クラス | ①継承? | ②自身でない? | ③このファイルで定義? | 結果 |
|---|---|---|---|---|
| `BaseCrawler` | ✅ (自分自身) | ❌ | ❌ | 除外 |
| `StaticCrawler` | ✅ | ✅ | ❌ (`src.framework.static`) | 除外 |
| `SampleScraper` | ✅ | ✅ | ✅ (`scripts.sites.sample`) | **✅ これだ！** |
| `Schema` | ❌ (無関係) | — | — | 除外 |

### 3.3 `issubclass()` — 継承関係の確認

```python
issubclass(SampleScraper, BaseCrawler)   # => True  (SampleScraper は BaseCrawler の子孫)
issubclass(BaseCrawler, BaseCrawler)     # => True  (自分自身も True になる！)
issubclass(Schema, BaseCrawler)          # => False (無関係)
```

> **注意:** `issubclass(A, A)` は `True` を返します。
> だから `obj is not BaseCrawler` のチェックが必要です。

### 3.4 `argparse` — コマンドライン引数の解析

```bash
python run_flow.py --site sample --url "https://example.com"
```

このような引数を受け取るための標準ライブラリです:

```python
import argparse

parser = argparse.ArgumentParser(description="説明文")

# 引数を定義
parser.add_argument("--site", required=True, help="サイト名")
parser.add_argument("--url",  required=True, help="対象URL")

# 解析して値を取得
args = parser.parse_args()
print(args.site)   # => "sample"
print(args.url)    # => "https://example.com"
```

- `required=True` → 省略するとエラーになる
- `help="..."` → `--help` で表示される説明文
- `parser.parse_args()` → `sys.argv` を解析して名前空間オブジェクトを返す

```bash
# 引数なしで実行するとエラーになる
$ python run_flow.py
error: the following arguments are required: --site, --url

# --help で使い方が表示される
$ python run_flow.py --help
usage: run_flow.py [-h] --site SITE --url URL
```

## 5. Step 1: import と sys.path — なぜ「おまじない」が必要か

```python
import sys
from pathlib import Path

root_path = Path(__file__).resolve().parent.parent
sys.path.insert(0, str(root_path))

from src.framework.base import BaseCrawler
```

### なぜ `sys.path` を操作するのか？

Python が `import src.framework.base` を実行するとき、
**`sys.path` に登録されたフォルダの中** から `src/framework/base.py` を探します。

```python
import sys
print(sys.path)
# ['/Users/eigou/.../scripts', '/usr/lib/python3.12', ...]
```

`run_flow.py` は `scripts/` フォルダにあるので、
Python のデフォルトの検索パスは `scripts/` です。
しかし `src/` は `scripts/` と同じ階層にはないため、見つかりません。

```
dso_system/           ← ★ ここを sys.path に追加したい
├── src/              ← import src.framework.base がここを見つけられない
│   └── framework/
└── scripts/          ← Python のデフォルト検索パスはここ
    └── run_flow.py
```

`Path(__file__).resolve().parent.parent` でプロジェクトルートを求め、
`sys.path.insert(0, ...)` でそこを検索パスの**先頭に**追加します。

```python
Path(__file__)           # bin/run_flow.py
  .resolve()             # /Users/.../dso_system/bin/run_flow.py （絶対パスに変換）
  .parent                # /Users/.../dso_system/scripts/
  .parent                # /Users/.../dso_system/               ← ここ！
```

> **`insert(0, ...)` の `0` の意味:**
> `sys.path` はリストで、Python は先頭から順番に検索します。
> `0` = 先頭に追加 → 最優先で検索される
> これにより、システムのライブラリと名前がかぶっても、自分のコードが優先されます。

## 6. Step 2: @task — クローラーの実行タスク

```python
@task(name="run_scraper", retries=0)
def run_scraper_task(site_name: str, url: str) -> str:
```

### デコレータの引数

- `name="run_scraper"` → Prefect ダッシュボードでの表示名
- `retries=0` → 失敗時の自動再試行回数（0 = 再試行しない）

> 将来 `retries=2` に変えれば、ネットワークエラーで失敗した場合に
> Prefect が自動で最大 2 回再試行してくれます。

### 動的インポート部分

```python
module_path = f"scripts.sites.{site_name}"
logger.info("モジュールをインポート: %s", module_path)

try:
    module = importlib.import_module(module_path)
except ModuleNotFoundError:
    raise ModuleNotFoundError(
        f"サイトモジュールが見つかりません: '{module_path}'\n"
        f"scripts/sites/{site_name}.py が存在するか確認してください。"
    )
```

`--site sample` が渡されると:

1. `module_path` = `"scripts.sites.sample"`
2. `importlib.import_module("scripts.sites.sample")` = `scripts/sites/sample.py` を読み込む
3. 存在しなければ `ModuleNotFoundError` — エラーメッセージを分かりやすく上書き

### クラス検出部分

```python
crawler_class = None
for name, obj in inspect.getmembers(module, inspect.isclass):
    if issubclass(obj, BaseCrawler) and obj is not BaseCrawler:
        if obj.__module__ == module.__name__:
            crawler_class = obj
            break
```

ここは [3.2 節](#32-inspect--クラスの中身を調べる) で詳しく解説済みです。
要約すると:

1. モジュール内の全クラスを列挙
2. 3 つの条件でフィルター:
   - `issubclass(obj, BaseCrawler)` → BaseCrawler の子孫か？
   - `obj is not BaseCrawler` → BaseCrawler 自身ではないか？
   - `obj.__module__ == module.__name__` → このファイルで定義されたか？
3. 条件を満たすクラスが見つかったら `break` で即座にループを抜ける

### 実行部分

```python
scraper = crawler_class()      # クラスをインスタンス化
scraper.execute(url)           # execute() を実行

return scraper.output_filepath or ""
```

`crawler_class` には `SampleScraper` クラス自体が入っています。
**Python ではクラスも変数に入れられます。** そして `()` をつけるとインスタンス化できます。

```python
# 普通の書き方
scraper = SampleScraper()

# 動的な書き方（変数にクラスが入っている）
crawler_class = SampleScraper     # ← クラスを変数に格納
scraper = crawler_class()          # ← 変数名() でインスタンス化
# 上2行は上1行と完全に同じ結果
```

`output_filepath or ""` は **短絡評価** です:

- `output_filepath` が `None` (0 件で CSV を作らなかった場合) → `""` を返す
- `output_filepath` が文字列 → その文字列を返す

## 7. Step 3: @flow — オーケストレーション

```python
@flow(name="scrape_site_flow")
def scrape_site_flow(site_name: str, url: str) -> str:
    logger = get_run_logger()
    logger.info("===== Flow 開始: site=%s, url=%s =====", site_name, url)

    output_path = run_scraper_task(site_name, url)

    logger.info("===== Flow 完了: %s =====", output_path)
    return output_path
```

Flow は「Task を呼び出す関数」です。現時点では:

```
scrape_site_flow()
  └── run_scraper_task()    ← 唯一の Task
```

**「Task が 1 つしかないのに、なぜ Flow で包むの？」**

理由は 2 つあります:

1. **Prefect の Management 機能が使える**
   - Flow = Prefect ダッシュボードでの管理単位
   - スケジュール実行、実行履歴、成功/失敗の追跡は全て Flow 単位

2. **将来の拡張に備えている**

   ```python
   # 将来のイメージ
   @flow(name="scrape_site_flow")
   def scrape_site_flow(site_name, url):
       output = run_scraper_task(site_name, url)         # Task 1
       transfer_to_nas_task(output)                       # Task 2 (将来追加)
       send_notification_task(site_name, output)          # Task 3 (将来追加)
   ```

   Flow の中に Task を足すだけで機能追加できます。
   `base.py` も `pipeline.py` も**一切変更不要**です。

## 8. Step 4: argparse — コマンドライン引数の受け取り

```python
if __name__ == "__main__":
    parser = argparse.ArgumentParser(
        description="Prefect Flow でクローラーを実行する"
    )
    parser.add_argument("--site", required=True, help="サイト名")
    parser.add_argument("--url",  required=True, help="クロール対象の URL")

    args = parser.parse_args()
    scrape_site_flow(site_name=args.site, url=args.url)
```

### `if __name__ == "__main__":` とは？

Python ファイルには **2 つの使い方** があります:

```python
# 使い方 1: 直接実行
$ python run_flow.py           # __name__ == "__main__" → True

# 使い方 2: 他のファイルから import
import run_flow                 # __name__ == "run_flow" → False
```

`if __name__ == "__main__":` の中に書いたコードは、
**ファイルを直接実行したときだけ**動きます。
他のファイルから `import` されたときには動きません。

これにより、将来 Prefect Deployment（自動スケジュール）から `scrape_site_flow` を
直接呼び出す場合に、`argparse` が邪魔しなくなります。

### 引数の流れ

```bash
python bin/run_flow.py --site sample --url "https://example.com"
```

```
sys.argv = ["bin/run_flow.py", "--site", "sample", "--url", "https://example.com"]
                                        ↓
parser.parse_args()
                                        ↓
args.site = "sample"
args.url  = "https://example.com"
                                        ↓
scrape_site_flow(site_name="sample", url="https://example.com")
```

## 9. 全体の実行フロー: 時系列で追う

```bash
python bin/run_flow.py --site sample --url "https://example.com"
```

### Phase 1: CLI → Flow

```
argparse が引数を解析
  site_name = "sample"
  url       = "https://example.com"
       │
       ▼
scrape_site_flow("sample", "https://example.com")
  │  Prefect が Flow の実行開始を記録
```

## 10. アーキテクチャ上の位置づけ

### レイヤー図

```
┌───────────────────────────────────────────────┐
│ Layer 3: 統括層 (Orchestration)                │  ← 今回のスコープ
│   bin/run_flow.py                          │
│   「いつ・何を・どの URL で実行するか」         │
│   使用技術: Prefect, argparse, importlib       │
├───────────────────────────────────────────────┤
│ Layer 2: コア基盤 (Framework)                  │  ← 前回のスコープ
│   base.py, pipeline.py, static.py, dynamic.py  │
│   「スクレイピングして CSV に保存する」         │
├───────────────────────────────────────────────┤
│ Layer 1: 共通部品 (Utilities)                  │
│   schema.py, normalizer.py                     │
│   「カラム定義と正規化ルール」                 │
└───────────────────────────────────────────────┘
```

### 各レイヤーが知っていること

```
統括層 (run_flow.py) が知っていること:
  ✅ BaseCrawler の存在（クラス検出に使う）
  ✅ execute(url) というメソッドがあること
  ✅ output_filepath というプロパティがあること
  ❌ Pipeline の存在を知らない
  ❌ normalizer の存在を知らない
  ❌ Schema のカラム名を知らない

コア基盤 (base.py) が知っていること:
  ✅ ItemPipeline
  ❌ Prefect の存在を知らない        ← ★ これが重要！
  ❌ argparse を知らない
  ❌ importlib を知らない
```

> **「知らない」= 変更の影響を受けない**
>
> Prefect を将来 Airflow に乗り換えたとしても、
> `base.py` や `pipeline.py` は**1行も変更不要**です。
> 変更するのは `run_flow.py` だけ。これが疎結合の最大のメリットです。

## 11. よくある疑問 Q&A

### Q1: `importlib` ではなく、普通の `import` ではダメなの？

ダメではないですが、サイトを追加するたびにコードを修正する必要があります:

```python
# ❌ 静的 import（サイト追加のたびに if 文が増える）
if site_name == "sample":
    from scripts.sites.sample import SampleScraper
    crawler_class = SampleScraper()
elif site_name == "tablog":
    from scripts.sites.tablog import TablogScraper
    crawler_class = TablogScraper()
elif site_name == "xxxxxxx":
    ...  # ← 新しいサイトを追加するたびに修正が必要

# ✅ 動的 import（サイトを追加しても run_flow.py の修正は不要）
module = importlib.import_module(f"scripts.sites.{site_name}")
# → scripts/sites/ にファイルを置くだけで自動的に使える
```

### Q2: `inspect` の代わりに、クラス名を引数で渡すのはダメ？

可能ですが、開発者の負担が増えます:

```bash
# inspect を使う場合（クラス名は自動検出）
python run_flow.py --site sample --url "https://..."

# クラス名も指定する場合（開発者がクラス名を覚えないといけない）
python run_flow.py --site sample --class SampleScraper --url "https://..."
```

「1ファイルに 1 クローラー」のルール（Convention）があるので、
ファイルさえ分かればクラスは自動で見つけられます。

### Q3: `@task(retries=0)` なら、リトライしないのに書く意味は？

明示的に「リトライしないことを選択している」という意思表示です。
スクレイピングは相手サイトにリクエストを送る処理なので、
安易にリトライすると相手サーバーに迷惑をかける可能性があります。

将来、安全と判断したら `retries=2` に変更するだけで自動リトライが有効になります。

### Q4: Flow も Task もただの Python 関数なのに、Prefect はどうやって管理しているの？

デコレータの仕組みです（[2.3 節](#23-デコレータ-flow-と-task) 参照）。

`@flow` を付けた関数を呼び出すと、Prefect は裏で:

1. 「Flow Run」レコードを Prefect サーバーに作成
2. 元の関数を実行
3. 成功/失敗/例外を記録
4. 実行時間を計測

これが全て **自動的に** 行われます。あなたのコードは普通の関数を書くだけ。

### Q5: Docker コンテナの中で実行する場合は何が違うの？

コード的には**何も違いません**。違いは環境変数だけです:

```
ローカル実行:
  PREFECT_API_URL が未設定 → Prefect はローカルの SQLite にログを保存
  → ダッシュボードなし（ターミナルにログが出るだけ）

Docker 実行:
  PREFECT_API_URL=http://server:4200/api  ← docker-compose.yml で設定済み
  → Prefect サーバーにログが送信される
  → ダッシュボード（http://localhost:4200）で実行履歴が見られる
```

コード側に `if docker:` のような分岐は一切ありません。
環境変数 1 つで挙動が変わる — これが **Convention over Configuration** です。

### Q6: 新しいサイトのスクレイパーを追加するときの手順は？

**`run_flow.py` の変更は不要です。** やることは:

1. `scripts/sites/newsite.py` を作る（`StaticCrawler` を継承、`parse()` を実装）
2. 実行コマンドを打つ:

```bash
python bin/run_flow.py --site newsite --url "https://newsite.com"
```

これだけです。`importlib` と `inspect` が自動でクラスを見つけてくれます。
